# 测试用例功能走查

将功能测试用例文档与项目源码进行系统性比对，识别差异、缺失实现和行为不一致。

## 概述

本 Skill 在测试用例文档与项目代码之间执行结构化审计。接受多种输入格式的测试用例，将其分解为逻辑模块，启动并行子 Agent 将每个模块与对应源码进行比对，最终生成**唯一一份差异点报告**。

**支持的输入格式：**
- XMind 思维导图（`.xmind`）
- Markdown 文档（`.md`，含结构化表格）
- PDF 文档（`.pdf`）
- URL（Figma 页面链接、共享文档链接、原始 Markdown URL）

**产出物（最终仅保留一份报告）：**
- 结构化测试用例 Markdown 文件（如 `WL6_TEST_CASES.md`）— 仅当输入非 Markdown 时需要
- **`<Project>_AUDIT_REPORT.md`** — **唯一最终产出**

> **关键约束：只生成一份最终报告。** 子 Agent 不得写入独立的 `*_COMPARISON.md` 文件，所有差异点直接汇编到最终报告中。报告仅列出差异点（含文件位置和具体描述），不列已覆盖项。每个差异点有唯一编号，用户可直接引用编号让 AI 精准修复。

---

## Workflow

### Step 1: 输入接受与校验

接受测试用例文档或链接，验证输入可访问且格式支持。

1. 若输入为本地文件路径，验证文件存在。
2. 若输入为 URL，使用 WebFetch 获取内容。
3. 若输入为 Figma 链接，使用 `get_figma_data` 提取文本节点并检查是否包含测试用例内容。
4. 若输入格式为 XMind，解析方案：
   - XMind 文件本质上是 ZIP 压缩包，内含 `content.json`
   - 使用 Python 解压并解析 JSON：`python3 -c "import zipfile, json; z = zipfile.ZipFile('file.xmind'); data = json.loads(z.read('content.json')); ..."`
   - 从 `rootTopic` 开始递归遍历 topic 树，提取 `title` 和 `children` 层级
   - 一级 topic → `##` 模块标题，二级 topic → `###` 子模块标题，三级/叶子 topic → 表格行
5. 不支持的格式给出清晰的提示信息。

**校验清单：**
- [ ] 文件存在且可读
- [ ] 内容包含可识别的测试用例结构（表格、列表或结构化标题）
- [ ] 至少能识别一个模块/分组
- [ ] 检测到编号方案（如 "S01"、"L01" 或顺序编号）

### Step 2: 解析为结构化 Markdown

将原始输入转换为统一的结构化 Markdown 文件，格式如下：

```markdown
# <Project Name> 测试用例

> **来源**: <原始来源标识>
> **生成日期**: <date>
> **用例总数**: <count>

## 目录

- [<模块 1>](#模块-1)
- [<模块 2>](#模块-2)

---

## <模块 1>

<可选的模块描述或流程说明>

### <子模块 1.1>

| 编号 | 用例/校验点 | 预期结果 |
|------|------------|---------|
| M01 | ... | ... |

### <子模块 1.2>
...
```

**解析规则：**

- **XMind 输入**：一级 topic → `##` 模块标题，二级 topic → `###` 子模块标题，叶子 topic → 表格行。若 topic 同时有子节点和备注，备注作为预期结果列。
- **Markdown 输入**：保留现有标题层级。检测并规范化表格为标准三列格式。若表格缺少预期结果列，尝试从上下文推断或标记为不完整。
- **PDF 输入**：按顺序提取文本，通过字号/加粗检测重建标题层级。表格提取最困难——使用行列位置启发式重建单元格。对列边界模糊的表格标记为需人工复核。
- **表格规范化**：确保每个测试用例表格恰好有三列。若源文件使用不同列名（如 "ID | 步骤 | 预期"），进行映射。若缺失某列，填入 "N/A" 并添加警告注释。
- **自动编号**：若源用例缺少稳定 ID，按模式 `{模块缩写}{子模块}_{序号}` 生成复合 ID。原始 ID 存在时优先保留。

将解析结果保存为 `<Project>_TEST_CASES.md`（项目根目录）。

### Step 3: 模块分解

分析已解析文档的标题结构，识别审计模块。

**算法：**
1. 扫描所有 `##` 标题行，每个定义一个**模块**。
2. 在每个模块内扫描 `###` 标题行，每个定义一个**子模块**。
3. 对每个子模块提取所有表格，每行为一个**测试用例**。
4. 构建模块树：
   ```
   模块 1 (标题: "## ...")
   ├── 子模块 1.1 (标题: "### ...")
   │   ├── 用例 1 (表格行)
   │   ├── 用例 2 (表格行)
   │   └── ...
   ├── 子模块 1.2
   │   └── ...
   └── ...
   模块 2
   └── ...
   ```

**自动生成模块 key**：从模块标题生成简短 slug（如 "启动与准入流程" → `startup-and-access`），作为整个审计流程中的模块标识。

**模块数量校验**：若少于 2 个模块，警告文档可能不适合分解。若超过 20 个模块，建议分批处理。

### Step 4: 模块→源码映射

为每个模块确定需要审计的源码目录。

**策略：**
1. 扫描项目源码树生成包映射。
2. 对每个模块名称/标题，使用语义匹配与包映射进行关联。
3. 以表格形式呈现拟议映射供确认：
   ```
   | 模块 | 建议源码路径 | 置信度 |
   |------|-------------|--------|
   | 启动与准入流程 | ui/startup/, ui/auth/ | High |
   | 注册登录流程 | ui/auth/ | High |
   ```
4. 若模块名无法匹配任何源码路径（如跨模块关注点 "贷超互通"），标记为 "distributed" 并扩大扫描范围。
5. 对每个映射目录，递归列出相关源文件（`.kt`、`.java`、`.xml`）。

**置信度启发式：**
- **High**：包名直接匹配模块关键词（如 "还款" → `repayment/`）
- **Medium**：部分语义匹配（如 "准入流程" → `startup/`）
- **Low**：无直接匹配，需广泛扫描
- **Manual**：需用户指定映射

映射关系内嵌在最终报告中，不额外生成 `MODULE_MAPPING.md` 文件。

### Step 5: 并行子 Agent 审计

每个模块启动一个子 Agent，使用标准化审计 prompt。最多 4 个并行子 Agent。

**子 Agent 审计 prompt 模板：**

---

```
你是模块 <MODULE_NAME>（key: <MODULE_KEY>）的审计 Agent。

## 测试用例

<插入该模块完整的测试用例表格，保留所有行>

## 待审计源码

<插入待扫描的源文件列表，含相对于项目根目录的完整路径>

## 审计指令

对每个测试用例，检查对应的源码并判定：

1. **COVERED（已覆盖）**：无需输出。
2. **PARTIAL（部分覆盖）**：行为部分实现但在关键细节上有差异。
3. **MISSING（缺失）**：描述的行为在扫描的源码中找不到对应实现。
4. **UNVERIFIABLE（无法验证）**：行为依赖服务端配置、外部 SDK 或跨模块逻辑，客户端无法确认。

## 输出要求

**只输出差异点（PARTIAL / MISSING / UNVERIFIABLE），不输出 COVERED 项。** 直接返回结果，不写入文件。

每个差异点格式：

| 编号 | 问题描述 | 预期结果 | 实际代码 | 文件位置 | 风险 |
|------|---------|---------|---------|---------|:--:|

编号格式：`M{模块序号}-{序号}`（如 M1-01, M2-03）。

已覆盖项一行带过："**已覆盖项**: xxx (N 项)"

## 检查维度

- **UI 元素**：描述的 UI 组件是否存在于布局 XML 中
- **文本/标签**：每条 toast 提示、错误提示、按钮文案是否精确匹配
- **输入校验**：校验规则是否实现（字符限制、格式检查、必填字段）
- **交互流程**：导航流程是否正确
- **状态处理**：所有状态是否处理（加载中、成功、空状态、错误）
- **配置依赖**：后端可配置功能是否有正确的开关控制
```

---

**执行参数：**
- 每个模块启动一个子 Agent。最多保持 4 个并行子 Agent。
- 若模块超过 4 个，分批顺序执行（每批 4 个）。
- 每个子 Agent 超时 600 秒。
- **子 Agent 不得写入 `*_COMPARISON.md` 文件**，所有结果直接返回给主流程。

### Step 6: 编译最终审计报告

所有子 Agent 完成后，将差异结果汇编为**唯一一份报告**。

**报告格式：**

```markdown
# <Project Name> 测试用例审计报告

> **测试用例来源**: <文件路径>
> **审计日期**: <date>
> **审计范围**: N 个功能模块，约 M 条校验点

---

## 执行摘要

<整体评估：核心流程状态 + 关键风险概述 + 差异点总数统计>

---

## 差异点清单

每个差异点可直接作为修复需求，编号格式 `M{模块}-{序号}`。

### 一、<模块1名称>

> 源码目录: <路径>

| 编号 | 问题描述 | 预期结果 | 实际代码 | 文件位置 | 风险 |
|------|---------|---------|---------|---------|:--:|
| M1-01 | 具体差异说明（含当前实现的文件名和方法） | 用例要求的正确行为 | 当前代码的实际行为 | `File.kt:method()` | Critical/High/Medium/Low |

**已覆盖项**: 简要列出（不展开详情）

---

### 二、<模块2名称>
...

---

## 附录：修复建议优先级

### P0 — 阻断性修复（N 项）
Mx-xx 描述...

### P1 — 高优先级修复（N 项）
...

### P2 — 中优先级修复（N 项）
...

### P3 — 低优先级修复（N 项）
...
```

**报告编写规则：**

1. **只列差异点，不列已覆盖项**。已覆盖项用一行简要说明（如 "已覆盖项: L01-L03/L06-L07（15 项）"）即可。
2. **每个差异点有唯一编号** `M{模块}-{序号}`，用户可直接说"修复 M3-01"让 AI 定位并修改。
3. **每个差异点含具体文件位置**（文件名 + 方法名），确保可直接定位源码。
4. **问题描述用中文说清楚差异是什么**，预期是什么，实际是什么。
5. **风险等级**四档：Critical（阻断性缺陷）、High（主要功能差异）、Medium（交互细节不一致）、Low（轻微差异）。
6. **附录含按优先级排序的修复编号清单**，方便用户按批次修复。
7. **所有子 Agent 的中间对比结果在报告编译完成后删除**，最终项目只保留 `<Project>_AUDIT_REPORT.md` 一份文件。

**清理步骤：**
- 报告写入后，立即删除所有子 Agent 可能写入的 `*_COMPARISON.md` 文件。
- 也不保留 `MODULE_MAPPING.md`（映射关系内嵌在报告附录中即可）。

## Guardrails

### 输入校验
- XMind 文件解析失败后重试 3 次，仍失败则要求用户提供其他格式。
- PDF 输入始终追加免责声明："PDF 表格提取可能存在格式错误；请手动复核关键单元格。"
- URL 输入超时 30 秒。若 URL 返回非文档响应（不含测试用例内容的 HTML 页面），拒绝并提示 "URL 未指向测试用例文档"。

### 范围边界
- 审计仅比对**行为**，不涉及代码质量、性能或安全——保持在功能正确性范围内。
- **禁止**在对比结果中直接建议代码修改。差异报告是观察层，修复需要经用户审核后的独立实现阶段。
- 审计期间**不得修改源码**。这是只读分析。
- 子 Agent 审计期间**不得访问外部资源**（API、数据库）。仅分析本地源文件。

### 精度约束
- 文本比对必须**精确**，包括空白、标点和大小写。单字符差异也要标记。
- 对于字符限制等校验规则，检查**实际约束值**，而不仅检查校验是否存在。
- **不得**仅凭文件名推测行为已实现，必须阅读实际代码。
- 对不确定的发现标记为 UNVERIFIABLE，不猜测。

### 子 Agent 管理
- 子 Agent 失败（超时或错误），重试一次。再次失败，在最终报告中将该模块标记为 "FAILED" 并备注需手动审计。
- 监控子 Agent 输出。若子 Agent 未产生任何发现（0 行），调查是否成功扫描了源码或存在路径解析问题。
- 若两个子 Agent 对重叠模块产生矛盾发现，标记两者供人工核对。

### 输出文件规则（强制）
- **只生成一份文件**：`<Project>_AUDIT_REPORT.md`。禁止保留 `*_COMPARISON.md`、`MODULE_MAPPING.md` 等中间文件。
- **Diff-only**：报告只列差异点（PARTIAL / MISSING / UNVERIFIABLE），不列已覆盖（COVERED）项的详情。已覆盖项一行带过。
- **每个差异点必须有唯一编号**：`M{模块序号}-{差异序号}` 格式，用户可引用编号直接让 AI 修复。
- **每个差异点必须含文件位置**：精确到 `文件名.kt:方法名()` 或 `文件名.xml`，确保可追溯。
- **子 Agent 不得写文件**：所有子 Agent 的结果直接返回给主流程，由主流程统一编译到最终报告中。

### 项目特定约束
- 对于 WL6 项目：已有 `WL6_TEST_CASES.md` 为基准文档。审计 WL6 时直接从该文件解析，除非用户提供了新的/更新的文档。
- 依赖项目的 `CLAUDE.md` 获取源码位置约定（如 XML 布局在 `res/layout/`，ViewModel 在 `ui/*/`）。
- 优先使用 ConstraintLayout、BaseRecyclerViewAdapterHelper 等 `CLAUDE.md` 中声明的项目约定来评估 UI 实现一致性。

### 反模式
- **不要重复解析**：若 `<Project>_TEST_CASES.md` 已存在且输入未变化，跳过 Step 1-2 直接使用已有文件。
- **不要跳过大模块的子 Agent**：一个 80+ 用例的模块仍然需要专门的子 Agent，不要将其合入其他 Agent。
- **不要忽略无法验证项**：始终在摘要中包含无法验证计数，并列出每项及其原因。
- **不要做逐行代码审查**：审计是测试用例驱动的，不是代码质量驱动的。聚焦于测试用例。
- **不要生成多份文件**：最终只保留 `<Project>_AUDIT_REPORT.md` 一份报告，中间文件必须清理。
- **不要在报告中列已覆盖项详情**：已覆盖项一行带过即可，报告的篇幅留给差异点描述。
